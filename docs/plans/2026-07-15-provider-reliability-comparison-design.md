# AI Watch Provider 可靠性趋势与对比设计

**日期：** 2026-07-15  
**状态：** 已确认

## 目标

把 AI Watch 已保存的结构化请求事件转化为可比较的 Provider 可靠性指标，帮助用户回答：

- 哪条 Provider 线路在当前时间窗内更稳定；
- 失败主要来自超时、过载、结果不匹配还是配置问题；
- 稳定性从何时开始恶化或恢复；
- 当前结论基于多少样本，是否受事件保留策略限制。

功能不得读取或返回 Prompt、API Key、代理明文、Webhook 或 CLI 原始输出。

## 方案选择

采用后端按需聚合现有结构化事件的方案。

- 相比前端聚合，后端可以统一分页、时间窗和统计口径，避免事件列表截断导致结果失真。
- 相比 Redis 预计算时间序列，按需聚合不需要新增写入路径、迁移或清理任务，适合当前数据规模。
- 后续若事件规模明显增长，可以保持 API 响应不变，把实现替换为预计算桶。

## 数据源

主要数据源为长期结构化 `request_end` 事件。该事件已经包含：

- `ProviderID`、`JobID` 和事件时间；
- `requestId`；
- 请求状态与分类结果；
- 请求耗时、退出码和安全错误类型；
- 已脱敏的 Job 元数据，包括 CLI、Provider 名称和模型。

聚合不依赖任务原始输出，也不使用短期请求日志缓存。

## API

新增：

```text
GET /api/reliability?range=24h|7d|30d
```

默认时间窗为 `24h`。其他值返回 400。

响应包含：

- 全局样本数、成功率、失败分布和 P95；
- Provider 指标列表；
- 等宽时间桶；
- 请求时间窗、事件保留天数和数据覆盖提示；
- 当前 Provider 是否仍存在，以及历史显示名称。

Redis 或事件存储不可用时返回 503。

## Provider 身份

聚合键使用 `CLI + Provider ID`：

- 普通 Provider：`codex:<provider-id>` 或 `claude:<provider-id>`；
- 当前配置：`codex:current` 或 `claude:current`。

这样可以避免 Codex 与 Claude 的空 Provider ID 相互混合。Provider 已删除时仍按事件中的历史名称展示，并标记为“历史配置”。

## 统计口径

### 请求结果

- `success`：成功请求；
- `timeout`：超时；
- `overloaded`：服务过载或限流；
- `unmatched`：返回成功但未匹配期望内容；
- `fatal`：配置、认证或其他不可恢复错误；
- `start_failed`：CLI 无法启动；
- `stopped`：用户或系统主动停止。

成功率计算为：

```text
success / (success + timeout + overloaded + unmatched + fatal + start_failed)
```

`stopped` 单独展示，不进入可靠性分母。

### 耗时

- 返回平均耗时；
- 至少 5 个有效耗时样本时返回 P95；
- 少于 5 个样本时 P95 为空，前端显示“样本不足”。

### 连续失败

请求按时间升序排列。成功会把连续失败计数归零；停止请求不改变连续失败序列。返回时间窗内的最大连续失败数。

## 时间桶

- `24h`：24 个一小时桶；
- `7d`：28 个六小时桶；
- `30d`：30 个一天桶。

每个桶返回开始时间、请求数、成功数、失败数、停止数、成功率和平均耗时。没有样本的桶返回零计数和空成功率，不使用零成功率误导用户。

## 前端

新增“可靠性”导航页，包含：

1. 时间窗选择器：24 小时、7 天、30 天；
2. 全局指标卡：请求数、成功率、P95、异常 Provider 数；
3. Provider 排名与对比表：成功率、样本数、失败分布、平均/P95、连续失败峰值和最近状态；
4. 趋势图：请求量与成功率时间桶；
5. 异常时段：失败密度最高的时间桶；
6. 数据覆盖提示：事件保留策略不足时明确说明统计范围受限。

页面默认按成功率降序排列；样本数不足的 Provider 排在完整样本之后。移动端改为卡片列表，核心指标保持可读。

## 错误与空状态

- 初次加载失败：显示错误状态和重试入口；
- 切换时间窗失败：保留上一次成功结果并显示非阻塞错误；
- 无请求事件：展示如何通过测活、保活或计划任务积累样本；
- 时间窗超过事件保留天数：显示部分覆盖提示；
- 历史 Provider：保留统计，但不提供启动任务操作。

## 测试

### 后端

- 结果分类与可靠性分母；
- 停止请求排除；
- CLI 与 Provider ID 隔离；
- 当前配置身份；
- 时间边界和等宽桶；
- 平均耗时与 P95；
- 连续失败峰值；
- 历史 Provider 标记；
- 非法范围和存储失败；
- 响应不包含 Prompt、密钥、代理或原始输出。

### 前端

- 默认 24 小时加载；
- 时间窗切换；
- Provider 排名和失败分布；
- 部分覆盖提示；
- 保留旧数据的刷新错误；
- 空状态；
- 320px 到桌面宽度无横向溢出。

## 完成标准

1. API 对三个时间窗返回稳定、可解释的聚合结果；
2. 页面能比较 Provider 并查看趋势与异常时段；
3. 数据覆盖不足时不会被表达成完整 SLA；
4. 不新增敏感数据持久化或返回；
5. Go 测试、前端构建、Playwright、Compose 配置检查全部通过。
