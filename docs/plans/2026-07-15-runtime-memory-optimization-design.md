# AI Watch 运行时内存优化设计

## 目标

在没有运行任务时，`ai-watch` 容器内存稳定低于 128 MiB，同时保留 Codex、Claude、MCP 和插件能力。任务执行期间允许按实际需要增长，但任务完成、超时、停止或失败后必须回落。

## 已确认事实

- Go 主服务 RSS 约 27 MiB。
- 高内存来自任务进程树，包括 Node 包装器、Codex 原生进程、MCP 服务和 Git 插件同步。
- 仅终止直接子进程无法可靠回收孙进程。

## 方案

1. 每个 CLI 任务运行在独立进程组。
2. 所有退出路径统一回收整个进程组：先取消上下文和发送 `SIGTERM`，短暂等待后使用 `SIGKILL` 兜底，并始终调用 `Wait`。
3. 任务结束后清理运行时临时目录。
4. 容器设置 `GOMEMLIMIT=96MiB`、`GOGC=80`，限制 Go 空闲堆保留，但不降低任务执行所需的容器硬上限。
5. 增加覆盖正常完成、超时、手动停止和启动失败的进程回收测试。

## 验收标准

- 连续执行任务后等待 30 秒，容器仅保留 `tini`、入口脚本需要的短生命周期进程和 `ai-watch`。
- 无残留 Node、Codex、Claude、MCP 或 Git 任务进程。
- 容器空闲内存低于 128 MiB。
- 现有 Go 测试、前端构建、容器健康检查通过。
