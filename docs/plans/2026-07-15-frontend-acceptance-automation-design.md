# AI Watch 前端验收自动化设计

**日期：** 2026-07-15  
**状态：** 已确认

## 背景

当前 Go 全量测试、前端生产构建和 Compose 配置检查均可通过。项目进度文档仍明确缺少主题切换、Redis 刷新、小屏供应商操作区以及多宽度、多主题页面的浏览器验收证据。

本轮先补齐这些既有验收缺口，再进入新功能开发。验收层不得依赖真实 Redis、Docker、Codex 或 Claude CLI，以保证本地和持续集成环境可以稳定复现。

## 方案选择

采用 Playwright 浏览器验收为主、少量组件测试为辅的组合方案。

- Playwright 负责真实浏览器交互、响应式布局、主题切换、关键状态和视觉证据。
- 组件测试仅用于难以稳定通过端到端流程触发的细粒度状态。
- 不采用纯截图像素比对，也不把仅有组件测试视为响应式和浏览器验收证据。

## 架构

在 `frontend` 中增加独立验收层。Playwright 启动 Vite 开发服务器，并在浏览器上下文中拦截 `/api/*` 请求，返回固定场景数据。测试不修改生产 API，也不要求后端服务处于运行状态。

共享场景数据覆盖：

- 健康状态和总览摘要；
- 当前配置、手填配置和只读 CC Switch Provider；
- Redis Key 列表、详情、刷新成功与失败；
- 计划任务、诊断信息和事件记录；
- 主题设置读取与保存。

测试辅助层统一管理 API Mock、视口、主题和页面导航，避免每个测试重复构造响应。

## 数据流

1. Playwright 启动 Vite 并创建浏览器上下文。
2. 测试在页面加载前注册 `/api/*` 路由拦截。
3. React 应用按照生产路径发起请求，Mock 层返回稳定场景数据。
4. 测试执行主题切换、刷新、导航、抽屉和响应式交互。
5. 测试断言可访问状态、布局边界和请求结果，并保存关键截图或 trace。

Redis loading 场景通过延迟响应实现；错误恢复场景先返回失败，再在重试请求中返回成功。主题保存场景同时验证前端即时切换和设置 API 请求。

## 浏览器覆盖

关键页面在以下宽度执行布局检查：

- 320px；
- 375px；
- 768px；
- 1024px；
- 1440px。

三套主题均覆盖关键页面。自动化断言包括：

- 页面不存在非预期横向溢出；
- 导航和主要操作可以访问；
- 供应商操作按钮在小屏下可见且可点击；
- 抽屉、确认框和编辑弹层不越出视口；
- 主题切换更新根节点状态并发送持久化请求；
- Redis 刷新期间 loading 可见，成功后数据更新，失败后可恢复。

## 视觉证据

截图用于人工复核关键区域，不以整页像素完全一致作为唯一通过条件。这样可以降低字体栅格化和平台差异造成的误报。

失败时保留：

- 当前页面截图；
- Playwright trace；
- 浏览器控制台错误；
- 失败步骤附近的 DOM 状态。

## 错误处理

- 未匹配的 API 请求应让测试立即失败，防止场景数据缺失被误判为产品空状态。
- 浏览器出现未处理异常或控制台 error 时，测试失败并输出来源。
- 延迟 Mock 必须有明确释放条件，避免测试永久等待。
- 截图和 trace 仅作为诊断证据，行为与布局断言仍是主要门禁。

## 命令与文档

前端增加统一命令：

```bash
npm run test:e2e
```

README 和项目进度文档记录安装、运行方式及验收范围。现有 `npm run build` 保持不变。

## 完成标准

以下证据必须全部成立：

1. `go test ./...` 通过；
2. `npm run build` 通过；
3. `npm run test:e2e` 通过；
4. `docker compose config --quiet` 通过；
5. 三主题和关键宽度均有自动化布局证据；
6. 主题切换、Redis 刷新和小屏供应商操作区均有真实浏览器交互证据；
7. README 与项目进度文档和实际命令一致。
