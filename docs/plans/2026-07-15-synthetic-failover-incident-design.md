# 合成测试、Provider 故障切换与事故中心设计

## 定位

AI Watch 继续作为单机个人 Provider 运维工具，不引入账号、租户和 RBAC。新增能力只作用于 AI Watch 自身的任务、计划和通知，不修改宿主机 Codex、Claude 或 CC Switch 配置。

## 一、合成测试场景

### 数据模型

`TestScenario` 包含：

- ID、名称、说明和启用状态。
- 适用 CLI：Codex、Claude 或两者。
- Prompt 和断言类型。
- 期望文本、正则表达式或 JSON 约束。
- 单次超时和创建、更新时间。

场景 Prompt 是用户明确创建的非敏感测试资产，允许加密存储或普通结构化存储；普通任务临时 Prompt 仍不持久化。界面必须提示不得放入密钥、个人信息或业务机密。

### 内置场景

- 基础可用性：只回复 `READY`。
- JSON 遵循：返回指定 JSON Object。
- 长上下文：验证较长输入能否被完整处理。
- 流式稳定性：验证流式调用是否能正常结束。
- 自定义文本或正则断言。

### 执行

- Job、Schedule 增加可选 `scenarioId`。
- 运行时解析场景并生成实际 Prompt 和断言，不把场景内容复制到长期请求事件。
- `request_end` 仅保存场景 ID、名称、断言类型和结果。
- 同一场景可批量运行多个 Provider，形成可比结果。

## 二、Provider 故障切换组

### 数据模型

`ProviderGroup` 包含 CLI、成员优先级、当前活跃成员、连续失败阈值、恢复成功阈值、策略模式和维护窗口。

策略模式：

- `advisory`：只生成切换建议。
- `automatic`：备用线路验证成功后，切换 AI Watch 计划使用的组内活跃 Provider。

### 安全边界

- 不修改宿主机配置。
- 自动切换前必须使用相同场景验证备用线路。
- 切换操作幂等并记录结构化事件。
- 没有健康备用线路时保持现状并升级告警。

### 自动模式执行约束

- 只有设置 `providerGroupId` 的计划会读取组内 `activeProviderId`。
- 自动切换只停止并重新调度绑定当前组的运行中计划，固定 Provider 计划保持原样。
- 用户手动发起备用验证与连续失败自动验证共用单一完成通道，避免旧快照覆盖刚完成的线路切换。
- 维护窗口内不启动评估、不切换线路；主线路连续成功达到恢复阈值后才回切。
- 切到备用后，由 Provider Group 自身按配置间隔使用同一合成场景探测主线路；探测频率不随绑定计划数量增长。
- 最近恢复探测时间和状态持久化，应用重启后继续恢复监测；恢复探测使用独立触发源，不递归进入故障切换评估。

## 三、事故中心

`Incident` 以 Provider 或 ProviderGroup 为主体，记录状态、开始和恢复时间、错误聚合、影响任务、关联 Request ID、确认、备注、静默和维护窗口。

相同主体在未恢复期间只维护一个开放事故。连续请求更新事故时间线，不重复创建主要告警。达到恢复条件后自动关闭并生成摘要。

## 实施顺序

1. 合成测试场景的数据模型、CRUD、任务选择和结果记录。
2. 多 Provider 场景对比。
3. ProviderGroup 与建议切换。
4. 自动切换和维护窗口。
5. Incident 聚合、通知降噪和事故页面。

## 验收标准

- 同一场景能对多个 Provider 产生可比较结果。
- 场景内容不会进入长期请求日志。
- 主线路失败后能验证备用线路并形成切换建议。
- 自动切换不修改宿主机配置。
- 连续失败只形成一个开放事故和一次主要通知。
- 恢复后事故自动关闭并保留时间线。
- 全量测试、浏览器验收和容器健康检查通过。
- 空闲内存持续低于 128 MiB。
