# Provider 建议式故障切换设计

## 目标

用户把同一 CLI 的主线路和备用线路组成故障切换组。主线路达到连续失败阈值后，AI Watch 使用同一个合成测试场景验证备用线路，并生成可审计的切换建议。首期不会自动修改计划任务或宿主机配置。

## 数据模型

`ProviderGroup` 保存 ID、名称、CLI、主 Provider、按优先级排列的备用 Provider、测试场景、连续失败阈值、建议冷却时间、启用状态和当前建议。

当前建议记录主线路失败 Request ID、建议备用 Provider、验证 Job/Request ID、原因、创建/更新时间和恢复时间。一个组同时最多存在一条开放建议。

## 执行流程

1. 主线路请求结束后检查连续失败次数。
2. 达到阈值且不在冷却期时，为该组取得独占验证锁。
3. 按优先级使用配置场景对备用 Provider 执行一次测活。
4. 首个验证成功的备用线路成为建议目标；没有健康备用线路则记录无可用备用的失败事件。
5. 主线路再次成功时关闭开放建议并记录恢复事件。

验证任务标记 `triggerSource=failover_validation`，不会递归触发新的故障切换评估。重复请求、并发请求和进程内重复检查都必须幂等。

## 安全边界

- 不修改 Codex、Claude、CC Switch 或宿主机配置。
- 不自动修改计划任务。
- 只引用 Provider ID 和 TestScenario ID，不保存凭证。
- 事件只保存脱敏 Request ID、Job ID、Provider ID 和建议原因。

## 界面

新增“故障切换”页面，支持创建、编辑、启停和删除组。列表展示主备拓扑、触发阈值、验证场景和当前建议；建议卡可打开主线路与验证请求详情。

## 验收

- CRUD 在 SQLite、Redis 均可往返。
- 连续失败只启动一次备用验证。
- 备用成功生成一条建议，重复失败不重复创建。
- 主线路恢复后关闭建议。
- 不健康备用不会改变任何计划。
- 全量 Go、浏览器、构建、容器和真实 API 验证通过。
