# 事故中心设计

## 目标

把 Provider 请求失败、可靠性告警和故障切换建议聚合成可追踪事故。相同主体在恢复前只保留一个开放事故，后续信号追加时间线而不重复创建主要通知。

## 模型

事故主体为 Provider 或 ProviderGroup。事故保存状态、严重程度、开始/更新时间、失败计数、错误分类、影响 Job、关联 Request ID、时间线、确认、备注、静默和恢复信息。

状态包括 `open`、`acknowledged`、`muted` 和 `resolved`。用户操作不删除历史事实；重新打开会追加时间线。

## 聚合

- Provider 非成功且非主动停止的 `request_end` 打开或更新事故。
- 重复 Request ID 不重复计数。
- 连续成功达到恢复阈值后自动关闭。
- 故障切换建议打开 ProviderGroup 事故；主线路恢复时关闭。
- 时间线只保存脱敏标识和结构化摘要。

## 通知

新事故发送一次主要通知，严重程度升级时允许再次通知，恢复后发送一次摘要。静默期间继续聚合但不发送通知。

## 页面

新增 `/incidents`，支持状态筛选、确认、备注、静默、关闭、重新打开和关联请求跳转。

## 安全与幂等

不保存 Prompt、凭证、代理密钥或完整响应。相同请求、相同用户操作和相同恢复信号必须幂等。
