# 测活与保活请求日志设计

**日期：** 2026-07-14  
**状态：** 已确认

## 目标

为每次测活和保活 CLI 调用生成准确、可回放的结构化请求日志。日志必须能够回答：何时请求、由谁触发、请求哪个目标、任务前解析到哪些 IP、使用何种代理、执行多久、返回什么状态、发生什么错误，以及对应的脱敏 stdout/stderr。

## 准确性边界

界面区分以下网络字段，不把推断信息冒充实际连接信息：

- 发起端 IP：从可信 HTTP 连接的 `RemoteAddr` 获取；计划任务标记为 `scheduler`。
- 目标地址：Provider Base URL 的脱敏域名和端口。
- DNS 解析 IP：每次 CLI 调用前由后端 Resolver 获取的全部结果。
- 代理模式与代理地址：Direct、默认 Mihomo 或自定义代理。

DNS 解析 IP 明确标记为“任务前解析结果”，不声明为实际 TCP 对端或代理出口 IP。当前系统不记录无法可靠关联的实际出口 IP。

## 数据模型

每次 CLI 调用生成唯一 `requestId`，记录：

- 任务 ID、尝试序号、模式和阶段。
- 触发来源：手动、批量、计划任务或恢复探测。
- 发起端 IP 或 `scheduler`。
- CLI、Provider、模型。
- 脱敏目标 URL、目标主机和端口。
- DNS IP 列表与 DNS 查询错误。
- 代理模式与脱敏代理地址。
- 开始时间、结束时间和单调时钟耗时。
- 状态：success、failed、timeout、stopped 或 start_failed。
- 退出码、分类结果、错误类型和脱敏错误信息。
- stdout/stderr 脱敏片段与截断标记。
- 下一次重试或保活执行时间。

## 事件流

复用现有任务事件和 SSE：

- `request_start`：生成 requestId、解析目标并记录开始信息。
- `request_log`：Runner 输出片段，携带 requestId 和流类型。
- `request_end`：记录最终状态、耗时、退出码、分类和错误。

每个 `request_start` 必须有唯一对应的 `request_end`，包括 CLI 启动失败、超时、用户停止和上下文取消。

## 保留策略

测活与保活统一写入 Redis 任务事件缓存：

- TTL 24 小时。
- 每任务最多 5,000 条。
- 每任务逻辑容量约 2 MiB。
- 超限按最旧事件裁剪。

任务摘要继续使用现有长期保留策略。请求日志缓存过期后，界面明确显示日志已过期，不伪造空结果。

## 安全边界

- 所有输出经过现有密钥、Token、代理凭证、路径和敏感参数脱敏。
- Prompt 不进入请求日志。
- 单次输出片段设置长度上限，截断时记录 `truncated=true`。
- 代理 URL 只展示脱敏端点。
- 不信任未配置代理环境下的 `X-Forwarded-For`。

## 后端实现

- 扩展任务启动元数据，记录 triggerSource 和 clientIP。
- 抽象可注入 DNS Resolver 和时钟，保证测试确定性。
- Runner Result 增加结构化执行时间、退出状态和错误分类所需字段。
- Manager 在每次 executor.Run 前后发布 request 事件，并把 output 关联到 requestId。
- 测活和保活都启用 JobEventStore 缓存。
- 历史 SSE 回放继续从 Redis 读取，无需新增独立命令控制台。

## 详情界面

任务详情增加“请求日志”页签：

- 顶部统计请求数、成功率、平均耗时、最近错误、DNS 与代理模式。
- 主列表展示时间、请求序号、阶段、状态、耗时、目标、DNS IP 和来源 IP。
- 展开行展示请求元数据、解析结果、代理说明、错误详情、退出码、下一次执行时间和脱敏日志。
- 实时任务通过 SSE 增量更新，历史任务通过 Redis 自动回放。
- 小屏使用请求卡片，保留时间、状态、耗时和错误入口。

## 验证

- start/end 一一对应且 requestId 唯一。
- 并发任务和连续保活不会串日志。
- 覆盖成功、失败、超时、停止和启动失败。
- 覆盖手动、批量、计划任务和恢复探测。
- 验证 DNS 多地址和查询失败。
- 验证 Redis TTL、行数和容量裁剪。
- 验证敏感数据不会进入 Redis、API 或前端。
- 运行 Go 全量测试、前端构建并重新部署容器。
