# 事件双视图与 CLI 请求详情设计

**日期：** 2026-07-14  
**状态：** 已确认

## 目标

事件记录支持列表和时间线自由切换。每次 CLI 调用形成可独立检查的结构化请求记录，尽可能展示后端能够可靠获得的 CLI、请求、网络、代理、执行结果和脱敏输出信息。

## 展示方式

- 列表视图作为首次默认值，适合扫描大量事件。
- 时间线保留现有视觉和事件顺序。
- 两种视图共用服务端筛选、分页和同一份查询结果。
- 用户选择保存到浏览器本地，下次进入恢复上次视图。
- 请求事件在列表中可展开，普通生命周期事件使用紧凑详情。

## 请求详情

每次 CLI 调用使用唯一 `requestId` 关联 `request_start`、`request_log` 和 `request_end`：

- 任务 ID、尝试序号、模式、阶段和触发来源。
- 发起端 IP；计划任务标记为 `scheduler`。
- CLI 类型、版本、Provider、模型和配置来源。
- 脱敏目标 URL、主机、端口、DNS 预解析 IP 和 DNS 错误。
- 代理模式及脱敏代理端点。
- 开始、结束、耗时、状态、退出码、分类结果和错误。
- stdout/stderr 脱敏片段、流类型和截断状态。
- 下一次重试或保活时间。

无法可靠获得的实际 TCP 对端或代理出口 IP 不做推断。DNS 地址明确标注为任务前解析结果。

## 安全与保留

- Prompt、API Key、Token、代理凭证、宿主路径和敏感参数不得返回前端或写入 Redis。
- 目标和代理地址只保存安全端点信息。
- 单片输出设长度限制并记录是否截断。
- 请求事件缓存保留 24 小时，每任务最多 5000 条、约 2 MiB。
- 缓存过期时界面明确提示，不用空数组伪装为没有执行过。

## Redis 刷新反馈

- 点击刷新后立即禁用刷新入口并显示旋转图标。
- Redis 工作区显示可见的加载遮罩和“正在刷新 Redis 数据”状态。
- 保留已有内容，避免刷新期间页面跳空。
- 成功显示 Toast，失败保留错误提示并移除加载态。

## 验证

- request start/end 一一配对，ID 唯一。
- 覆盖成功、失败、超时、停止和启动失败。
- 覆盖手动、批量、计划任务和恢复探测来源。
- 验证敏感信息不会进入 Redis、API 或页面。
- 验证列表/时间线共享筛选和分页并记住选择。
- 验证 Redis 刷新期间 loading 可见，成功和失败都能结束 loading。
- 运行 Go 全量测试、前端生产构建和容器健康检查。
