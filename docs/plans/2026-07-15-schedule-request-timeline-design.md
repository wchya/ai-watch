# 计划任务请求时间线设计

**日期：** 2026-07-15  
**状态：** 已确认

## 目标

在计划任务列表中选择日志入口后，查看该计划任务 ID 产生的全部现存请求日志与脱敏后的供应商返回信息。

## 数据关联

计划任务启动 Job 时，将 `scheduleId` 保留在运行上下文，并写入该 Job 产生的持久化事件。事件查询 API 支持按 `scheduleId` 过滤，避免只依赖计划上的最近一次 `lastJobId`。

## 交互

计划任务行的日志按钮打开独立日志抽屉。日志以请求为单位形成时间线，按请求开始时间倒序排列，最新请求在前。每条记录展示状态、时间、Provider、尝试次数、耗时和 Job ID；展开后展示脱敏供应商返回、错误以及安全请求元数据。

时间线支持分页、刷新与状态筛选。空状态区分“尚未运行”和“日志已被保留策略清理”。

计划处于排队、启动或运行状态时，规则行使用高可见运行态并锁定所有可能改变或重复启动该规则的入口，只保留日志查看。操作按钮按日志、启停、编辑和删除使用不同语义颜色。

Codex 输出在进入长期事件前剥离 CLI 版本、工作目录、沙箱、session、用户输入和 token 统计，只保存最终助手回答。失败或超时优先展示脱敏错误；没有有效正文时展示明确的超时或未识别响应说明。

## 安全与保留

沿用现有事件保留策略。Prompt、API Key、Token、认证头、代理凭证和宿主路径不得进入 API 或页面。供应商返回仅使用已有脱敏、截断后的响应摘要。

## 验证

- 同一计划多次运行产生的请求均可按 `scheduleId` 查询。
- 不同计划和手动任务不会混入时间线。
- 时间线默认最新在前，分页无重复或遗漏。
- 成功、失败、超时和无返回信息均有清晰展示。
- 前端构建、Go 测试和浏览器交互测试通过。
