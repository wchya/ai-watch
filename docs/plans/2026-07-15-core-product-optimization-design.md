# AI Watch 核心产品全量优化设计

## 目标

围绕“选择 Provider → 发起测试或保活 → 获得可信结论 → 定位失败原因 → 自动化计划与通知”的主链路重构应用，同时提升执行稳定性与操作体验。

## 当前问题

- 手动任务、批量任务和计划任务存在相似但分散的启动与停止语义。
- Job、尝试和实际请求的边界不够明确，日志、事件与可靠性页面展示同一事实的不同切片。
- 活动任务使用 SSE，而列表、计划和全局数据使用多套轮询与延迟刷新，容易短暂不一致。
- 失败信息分散在任务详情、计划日志、事件、可靠性和诊断页面，用户需要自行拼接原因。
- 调度、诊断和旧数据迁移测试存在不稳定或环境耦合问题。
- 部分前端请求失败被静默忽略，旧数据显示时缺少明确提示。

## 核心模型

统一采用以下层级：

```text
Provider
  └─ Job
      └─ Attempt
          └─ Request
              ├─ 配置摘要
              ├─ CLI 与供应商输出
              ├─ 分类结果
              ├─ 结构化错误
              └─ 时间与耗时
```

每个实际请求拥有稳定 Request ID。手动任务、计划任务、事件、日志和可靠性统计都引用相同的请求事实。

## 第一阶段：可信执行内核

- 统一手动、批量和计划任务的启动命令模型。
- 明确停止、超时、重试、恢复探测、计划抢占和并发锁语义。
- 统一结构化错误：阶段、类型、消息、退出码、是否可重试和供应商响应摘要。
- 修复调度优先级、诊断接口和迁移测试的不稳定性。
- 保持任务进程组回收与低空闲内存目标。

## 第二阶段：统一观测链路

- 请求事实成为计划时间线、事件记录和可靠性统计的共同数据源。
- 任意入口都能打开同一个请求详情。
- 请求详情统一展示结论、供应商响应、执行上下文和时间线。
- 长期保存脱敏结构化摘要；完整 CLI 日志继续有界、短期保存。

## 第三阶段：核心体验重组

- 首页升级为可用性与行动中心，突出异常线路、运行任务和待处理计划。
- Provider 行直接提供测试、保活、最近请求和创建计划。
- 可靠性建议可直接触发测试替代线路或暂停异常计划。
- 系统诊断只负责基础设施问题，业务请求问题回到请求详情。
- 旧数据、离线状态和刷新失败必须显式标识。

## 第四阶段：性能与工程质量

- SSE 负责活动任务；轮询只作为低频兜底，并受页面可见性控制。
- 合并重复 API 请求并支持取消过期请求。
- 可靠性计算使用有界查询或增量聚合，避免全量事件加载。
- 建立确定性调度测试、核心集成测试、浏览器主流程测试和容器冒烟测试。
- 删除失效兼容分支、重复状态和遗留样式覆盖。

## 错误与状态原则

- 不静默吞掉会导致页面过期或操作失败的错误。
- 状态以服务端请求事实为准，前端乐观状态只用于短暂交互反馈。
- 同一 Request ID 的终态只能写入一次，重复事件必须幂等。
- 用户看到的错误必须回答：哪里失败、为什么、供应商返回什么、能否重试、下一步做什么。

## 验收标准

- 从选择 Provider 到获得结论不超过三个主要操作。
- 任一失败请求可在一个详情页查看完整原因。
- 手动任务与计划任务对相同配置产生一致结果。
- 后端测试连续多次稳定通过。
- 停止、重试和计划启动后的页面状态及时一致。
- 空闲内存持续低于 128 MiB。
- 核心页面不产生明显重复请求。
