# AI Watch 核心产品收拢与深度减重设计

## 核心流程

唯一核心流程为：配置 Provider → 手动测活或场景验证 → 创建计划 → 查看异常 → 处置或切换线路。

重构允许删除低频页面、API、存储模型和历史兼容分支，但不得破坏 Provider、计划、请求、事故、SLO、故障切换和通知配置等核心数据。

## 产品结构

一级入口由 15 个收拢为 7 个：总览、Provider、验证中心、自动化、稳定性、事件、设置。

- Provider：发现的 Provider 与手动 Provider；
- 验证中心：测试场景、批量 Provider 对比、对比历史；
- 自动化：计划任务、故障切换和维护窗口；
- 稳定性：可靠性、SLO、事故和复盘；
- 事件：普通事件、请求日志和请求详情；
- 设置：任务默认值、通知、通知路由、诊断和保留策略。

旧一级路径重定向到对应中心，不保留第二套页面实现。

## 删除范围

1. 删除供应商示例的页面、弹窗、API、Store、类型和持久化集合；
2. 删除 Redis 管理页面和 `/api/redis/*` 管理 API；Redis 只作为内部存储，通过诊断显示健康与容量；
3. 删除“发送状态汇总”独立通知操作；
4. 删除计划请求时间线的旧 Job 回退兼容；
5. 删除 Provider Group 编辑器中的旧维护截止字段；
6. 删除已经完成迁移、无法再产生可靠价值的旧 UI 字段兼容。

## 前端结构

路由注册表成为 path、标题、图标、侧边栏与页面渲染的唯一来源。`App.tsx` 只保留应用外壳、全局任务抽屉和请求详情协调。

领域中心内部使用标签页。通用异步读写、确认、错误、空状态、抽屉和对话框逐步抽为公共组件与 hooks。全局 CSS 拆分并删除旧固定色与重复覆盖。

## 后端结构

API 入口按 Provider、验证、自动化、稳定性、事件、设置拆分路由函数。统一 JSON 解码、对象不存在、审计事件、分页和 CRUD 错误响应。

场景、Provider Group、事故、复盘和通知渠道等文档型实体逐步使用统一集合存储，保留 SQLite 迁移源和 Redis 生产存储的既有边界。

Jobs Manager 保留现有领域文件，但主 Manager 只负责生命周期和协调；调度、通知、事故、故障切换和保留逻辑不再向主文件堆积。

## 数据与迁移

- SQLite 新增清理迁移，删除 `provider_examples`；
- Redis 启动时删除供应商示例集合；
- 不删除 Provider、计划、事件、场景、对比、事故、复盘、SLO、故障切换、维护窗口和通知渠道；
- 旧页面 path 重定向到新领域中心；
- 已有深链接 `/requests/:id` 保持有效。

## 成功标准

- 一级导航不超过 7 项；
- 核心动作从总览最多 3 次点击到达；
- Redis 管理和供应商示例代码完全删除；
- `App.tsx` 低于 350 行，`server.go` 低于 500 行；
- 页面不再重复实现基础 loading、错误和延迟刷新；
- 核心 E2E 覆盖 Provider、验证、自动化、稳定性、事件和设置；
- 全量 Go、前端构建、E2E、容器和真实数据只读验收通过；
- README、架构与运维文档和代码一致。
