# Redis 管理界面设计

**日期：** 2026-07-14  
**状态：** 已确认

## 目标

为 AI Watch 增加一个美观、响应式的 Redis 管理界面。该界面可以浏览整个 Redis 数据库中的 Key，查看常用数据类型的内容和元信息，并执行缓存预热、更新、TTL 调整、重命名和删除等受控操作。

管理能力仅依赖现有的本机访问边界，不新增登录鉴权。服务继续绑定 `127.0.0.1`，所有危险写操作都要求二次确认。

## 范围

首版支持以下 Redis 数据类型：

- String
- Hash
- List
- Set
- ZSet

其他类型可以显示 Key 名、类型、TTL、元素数量和内存占用等基础信息，但不提供内容编辑。

页面能够浏览整个 Redis 数据库，而不仅限于 `ai-watch:*` 命名空间。首版不提供任意 Redis 命令控制台，也不提供 `FLUSHDB` 或 `FLUSHALL`。

## 方案选择

采用受控 Redis 管理 API。后端提供固定的白名单操作，前端只能提交结构化请求，不能透传任意 Redis 命令。

相比命令代理，该方案可以限制阻塞命令和破坏性操作，控制响应大小，并为 AI Watch 自有数据提供预热、校验和回滚能力。相比业务 API 与通用浏览器混用，该方案的交互规则更一致，也更容易测试。

## 后端架构

新增独立 Redis 管理服务，直接持有 Redis 客户端，但不扩展通用 `store.Store` 接口。管理服务负责：

- Redis 运行状态和统计信息。
- 使用 `SCAN` 分页检索 Key。
- 获取 Key 类型、TTL、编码、元素数量和内存占用。
- 分页读取 String、Hash、List、Set、ZSet 内容。
- 对五种类型执行结构化更新。
- 设置 TTL、移除 TTL、重命名和删除 Key。
- 触发 AI Watch 应用级缓存预热。
- 对 `ai-watch:*` 写操作执行备份、预热和失败回滚。

所有列表操作使用 `SCAN` 或对应的游标、范围命令，禁止使用 `KEYS`。后端限制请求体、字段长度、返回元素数量和单值大小，避免 Redis 或浏览器被超大响应拖垮。

## API 设计

新增 `/api/redis` 路由组：

- `GET /api/redis/overview`：连接状态、版本、Key 总数、内存、命中率和过期统计。
- `GET /api/redis/keys`：按 pattern、类型、cursor 和 limit 扫描 Key。
- `GET /api/redis/keys/detail`：读取指定 Key 的元信息和分页内容。
- `PUT /api/redis/keys/value`：更新 String 或集合成员。
- `PUT /api/redis/keys/ttl`：设置 TTL 或切换为持久化。
- `POST /api/redis/keys/rename`：重命名 Key。
- `DELETE /api/redis/keys`：删除 Key。
- `POST /api/redis/prewarm`：重新加载 AI Watch 的进程内快照。

写请求携带最近读取到的版本指纹。后端在执行前重新计算指纹，发现内容已经变化时返回冲突，避免覆盖其他进程的新数据。

危险操作需要提交明确确认信息。删除、覆盖和重命名必须携带完整 Key 名作为确认值，后端不能只依赖前端对话框。

## AI Watch 数据一致性

直接修改 `ai-watch:*` Key 可能影响设置、供应商、计划任务和其他进程内快照。此类操作采用以下流程：

1. 读取并保存旧类型、旧内容和旧 TTL。
2. 执行受控 Redis 修改。
3. 重新加载全部 AI Watch 应用快照。
4. 若解析或预热失败，恢复旧值和旧 TTL。
5. 再次加载旧快照，并向前端返回回滚结果。

预热操作由互斥锁串行执行，避免并发请求交错替换内存状态。非 `ai-watch:*` Key 不触发应用预热。

若回滚本身失败，API 返回严重错误。前端持续显示醒目告警，提示 Redis 与应用内存可能不一致，并建议重启服务重新加载状态。

## 页面结构

侧边栏新增“Redis 管理”入口。页面延续现有深色控制台视觉语言，使用清晰的色彩层级、紧凑数据表格、固定操作栏和即时反馈。

### 顶部概览

展示连接状态、Redis 版本、Key 总数、已用内存、命中率和过期 Key 数量，并提供“应用缓存预热”主按钮。

预热完成后展示设置、摘要、供应商、计划任务等每类快照的数量、耗时和成功状态。

### Key 浏览器

左侧区域支持：

- Key 关键词和通配 pattern。
- 数据类型筛选。
- 每页数量。
- 游标翻页。
- 类型、TTL 和内存占用摘要。

筛选输入采用防抖请求。TTL 临近到期时使用渐变告警，不同数据类型使用稳定且可区分的标签色。

### Key 详情

右侧展示完整 Key 名、类型、TTL、编码、元素数量、内存占用、版本指纹和内容。

- String：文本编辑，合法 JSON 可格式化显示。
- Hash：字段表格，可新增、修改和删除字段。
- List：按索引分页，可追加、插入、修改和删除。
- Set：成员分页，可添加和移除成员。
- ZSet：成员与 Score 表格，可排序、新增和修改。

未知类型只显示元信息和只读提示。

### 危险操作

详情页底部提供 TTL 修改、持久化、重命名和删除操作。删除、覆盖、重命名和集合清空要求输入完整 Key 名确认。

页面使用 Toast 展示成功结果，使用可展开错误详情展示服务端错误。加载时使用骨架屏，无结果时提供明确空状态。

### 响应式布局

宽屏使用 Key 列表与详情双栏布局。窄屏先显示 Key 列表，详情使用抽屉或独立面板打开，表格内容允许局部横向滚动，避免页面整体溢出。

## 数据流

页面首次进入时并行加载 Redis 概览和第一批 Key。选择 Key 后先读取类型和元信息，再根据类型分页加载内容。

集合更新使用对应的 Redis 原子命令或 Lua，不使用无保护的“删除后重建”。更新时尽量保留原 TTL。

String 或集合内容过大时，后端只返回元信息、截断标记和安全预览，不允许超大值直接进入浏览器编辑器。

所有内容以纯文本渲染。AI Watch 已加密保存的敏感数据只显示密文摘要，不通过管理接口解密。

## 错误处理

API 和前端明确区分：

- Redis 连接失败。
- Key 不存在或读取期间被删除。
- Key 类型发生变化。
- 版本指纹冲突。
- 数据或响应过大。
- 参数校验失败。
- 应用预热失败但回滚成功。
- 应用预热和回滚同时失败。

所有 API 响应继续设置 `Cache-Control: no-store`。
