# 全系统写操作幂等设计

**日期：** 2026-07-15  
**状态：** 已确认

所有前端写请求携带 `Idempotency-Key`。服务端以请求方法、完整路径和请求体的 SHA-256 作为指纹：同 Key 同指纹只执行一次并回放原始状态码、Content-Type 与响应体；同 Key 不同指纹返回 `409 idempotency_conflict`。并发重复请求等待首个执行者完成，超时返回 `idempotency_in_progress`。

生产环境使用 Redis `SET NX` 原子抢占，记录保留 24 小时；没有 Redis 的测试或迁移环境使用进程内回退。异常执行会写入可回放的 500 结果，避免 Key 永久停留在 pending。没有提供有效 Key 的旧客户端保持兼容，但 Web UI 的所有写操作默认生成 Key。

验证覆盖并发重复创建只产生一个资源、响应完全一致、相同 Key 不同请求冲突、Redis 抢占与完成回放，以及现有全量测试。
