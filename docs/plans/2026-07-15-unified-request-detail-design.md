# 统一请求详情与深链接设计

## 目标

让用户从首页、Provider、事件记录、计划日志和可靠性页面打开同一个请求事实，并通过 `/requests/:requestId` 刷新、分享和使用浏览器前进后退。

## 数据边界

请求详情只聚合现有脱敏 `request_start`、`request_end` 结构化事件。请求输出不新增长期存储，Prompt、API Key、Token、代理凭证、宿主路径和完整供应商响应不进入接口。

服务端增加按 `requestId` 的事件过滤，并提供 `GET /api/requests/:requestId`。响应包含请求标识、任务/计划/Provider 关联、CLI/模型、触发来源、脱敏目标和代理摘要、时间、耗时、状态、退出码、分类、错误阶段、错误类型、是否可重试、返回摘要和下一次尝试时间。

## 页面与导航

- 路由：`/requests/:requestId`。
- 页面采用独立详情布局，不使用遮罩层。
- 返回按钮优先调用浏览器历史；直接访问时返回请求日志页。
- 事件请求日志与计划请求时间线提供“打开完整详情”。
- Provider 的“最近请求”先进入带 Provider 筛选的请求日志；具体请求再进入详情。
- 可靠性建议和首页仅在拥有具体 requestId 时提供详情入口，避免推断不存在的关联。

## 错误处理

- 找不到请求时返回 404 `request_not_found`。
- 事件保留期已过时页面明确提示“请求详情已过期或被清理”。
- start/end 不完整时仍返回可用字段，并标记请求正在执行或记录不完整。
- 错误详情回答失败阶段、错误类型、是否可重试和建议动作。

## 验证

- SQLite 与 Redis 均能按 requestId 精确过滤。
- API 聚合 start/end，不泄露敏感字段。
- 深链接可直接访问、刷新和浏览器返回。
- 事件与计划入口跳转到同一请求详情。
- 三套主题、320px 和桌面布局通过浏览器验收。
